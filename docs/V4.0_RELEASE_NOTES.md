# v4.0 Release Notes — Split Search Architecture

**Date**: February 27, 2026  
**Status**: Deployed and verified

---

## Problem

Two related issues with the audience builder search at scale (~240K records):

1. **Keyword searches timed out**: The original `search_free_data_builder` function tried to do both data retrieval and counting in one call. For keyword searches (JSONB ILIKE), this exceeded the 8-second `authenticated` role statement timeout.

2. **EXPLAIN estimates were wildly inaccurate**: To work around timeouts, the count function used Postgres `EXPLAIN (FORMAT JSON)` planner estimates for keyword searches. For "CEO", this returned **771** when the actual count was **21,282** — off by 27x.

---

## Solution

### Split Architecture with Parallel RPC Calls

Replaced the single `search_free_data_builder` RPC (for frontend use) with two independent functions called in parallel:

```
Frontend (useFreeDataSearch.ts)
  │
  ├── search_free_data_results(filters, limit, offset)
  │   └── Returns: entity_data, entity_external_id
  │   └── Timeout: 15s (SECURITY DEFINER)
  │   └── Speed: ~1-2s
  │
  └── search_free_data_count(filters)
      └── Returns: total_count, is_estimate
      └── Timeout: 30s (SECURITY DEFINER)
      └── Speed: ~3-5s for keywords, instant for unfiltered
```

Both use `SECURITY DEFINER` to bypass the 8-second role-level timeout.

### Three Phases of the Fix

| Phase | What Changed | Result |
|-------|-------------|--------|
| 1. Split functions | Separated data retrieval from counting into independent RPCs | Results display in ~1-2s regardless of count complexity |
| 2. Remove EXPLAIN estimates | Replaced `EXPLAIN (FORMAT JSON)` with exact bounded count for all filtered queries | CEO count: 771 → 21,282 |
| 3. Unified bounded count | All filtered queries use `SELECT count(*) FROM (... LIMIT 100001) _sub` | Consistent accuracy, `is_estimate = false` for all filtered searches |

### Count Strategy

| Scenario | Method | `is_estimate` |
|----------|--------|---------------|
| Filtered (any filter including keywords) | Exact bounded count (LIMIT 100001) | `false` |
| Unfiltered (base entity count) | `pg_class.reltuples` | `true` |
| Timeout / error | Fallback: count = 0 | `true` |

## Performance Characteristics

| Search Type | Results Time | Count Time | Count Accuracy |
|-------------|-------------|------------|----------------|
| No filters | ~1s | Instant | Estimate (pg_class) |
| Keyword: "CEO" | ~1-2s | ~3-5s | Exact: 21,282 |
| Gender: Male | ~1s | ~1-3s | Exact |
| Company Size: 10000+ | ~1s | ~1-2s | Exact |
| Combined filters | ~1-2s | ~2-5s | Exact |

---

## Backward Compatibility

`search_free_data_builder` (37 parameters) is **retained** in the database for backward compatibility but is no longer called from the frontend. All new frontend calls use the split functions.

---

## Verification

```sql
-- Keyword count (was 771, now 21,282)
SELECT total_count, is_estimate FROM public.search_free_data_count(
  p_entity_type := 'person', p_keywords := ARRAY['CEO']
);
-- Expected: ~21,282, is_estimate = false

-- Results function works
SELECT count(*) FROM public.search_free_data_results(
  p_entity_type := 'person', p_limit := 10, p_offset := 0
);
-- Expected: 10
```
