# V3.7 Release Notes

**Date:** February 18, 2026  
**Status:** Stripe checkout flow working end-to-end  
**Known Issue:** Occasional post-checkout logout (see below)

---

## Summary

This release fixes the end-to-end Stripe subscription checkout flow. Three independent root causes were identified and resolved in sequence.

| # | Bug | Root Cause | Fix |
|---|-----|------------|-----|
| 1 | White screen after Stripe redirect | `if (!user) return null` in `ProtectedRoute` | Removed the guard; spinner covers transient null |
| 2 | "Verifying your payment..." infinite spinner | Checkout logic embedded inside `ProtectedRoute` conflicted with auth loading cycles | Moved verification to new dedicated `/checkout-success` route |
| 3 | Immediate logout after successful payment | `onAuthStateChange` fired transient `SIGNED_OUT` and cleared user state | Guard against `SIGNED_OUT`; handle `INITIAL_SESSION` |

## Root Cause 1 — White Screen After Stripe Redirect

**File:** `src/components/ProtectedRoute.tsx`

### What happened

The component had `if (!user) return null` which rendered a completely blank screen whenever `user` was briefly `null` during Supabase's auth recovery sequence. After a cross-origin redirect from Stripe, Supabase emits a transient sequence:

```
SIGNED_OUT → INITIAL_SESSION
```

During this window, `user` momentarily became `null`, causing the component to return `null` (blank screen) instead of showing a loading spinner.

### Before / After

```
BEFORE:
  Stripe redirect → user = null (transient)
  → ProtectedRoute: if (!user) return null
  → White screen (blank page, no spinner, no redirect)

AFTER:
  Stripe redirect → user = null (transient)
  → ProtectedRoute: if (loading || (profileLoading && !profile)) → spinner shown
  → INITIAL_SESSION fires → user restored → profile loaded → Dashboard ✓
```

### Fix

Removed the `if (!user) return null` guard entirely. The existing `useEffect` already handles genuine unauthenticated users by calling `navigate('/auth')`. The spinner condition `if (loading || (profileLoading && !profile))` now correctly covers the transient window.

## Root Cause 2 — "Verifying your payment..." Infinite Spinner Deadlock

**Files:** `src/components/ProtectedRoute.tsx`, `src/pages/CheckoutSuccess.tsx` (new), `src/App.tsx`, `supabase/functions/create-checkout/index.ts`

### What happened

The checkout verification logic was embedded directly inside `ProtectedRoute`. The component maintained local state `isVerifying` which was set to `true` when `?checkout=success` was detected in the URL. The problem: `profileLoading` toggled `true/false` repeatedly as the profile loaded and re-loaded, causing the component to re-render in a way that prevented `setIsVerifying(false)` from ever persisting. The spinner was permanently stuck on screen.

### Before / After

```
BEFORE:
  /dashboard?checkout=success → ProtectedRoute detects param
  → setIsVerifying(true)
  → profileLoading toggles true/false (re-renders)
  → setIsVerifying(false) never sticks
  → Spinner shown forever ∞

AFTER:
  Stripe success_url → /checkout-success (unprotected route)
  → CheckoutSuccess: invoke check-subscription
  → fetchProfile()
  → toast("Payment confirmed! Welcome to Vrelly...")
  → navigate('/dashboard', { replace: true }) ✓
```

### Fix

Moved all checkout verification to a new, dedicated, unprotected route `/checkout-success`. This page:

1. Gets the current session via `supabase.auth.getSession()`
2. Calls the `check-subscription` edge function with the user's access token
3. Calls `fetchProfile()` to refresh the profile in the auth store
4. Shows success toast: *"Payment confirmed! Welcome to Vrelly — your credits are ready."*
5. Navigates to `/dashboard` with `replace: true`

The Stripe `success_url` in `create-checkout/index.ts` was updated:
```
BEFORE: ${origin}/dashboard?checkout=success
AFTER:  ${origin}/checkout-success
```

The `/checkout-success` route in `App.tsx` was registered as unprotected (no `ProtectedRoute` wrapper). `ProtectedRoute` was restored to a simple auth + subscription guard with zero checkout logic.

## Root Cause 3 — Immediate Logout After Successful Payment

**File:** `src/components/AuthProvider.tsx`

### What happened

After Stripe redirects back to the app (cross-origin), Supabase's `onAuthStateChange` fires a transient `SIGNED_OUT` event as part of its normal session recovery handshake. The listener was calling `setUser(session?.user ?? null)` unconditionally, which set `user = null`. This caused `ProtectedRoute` to fire `navigate('/auth')` before the subsequent `INITIAL_SESSION` event could restore the session.

```
SIGNED_OUT → user = null → navigate('/auth') → LOGGED OUT
              (happens before INITIAL_SESSION can recover)
```

### Before / After

```
BEFORE:
  Stripe redirect → onAuthStateChange('SIGNED_OUT', null)
  → setUser(null)
  → ProtectedRoute useEffect: !user → navigate('/auth')
  → User lands on /auth (logged out) ✗

AFTER:
  Stripe redirect → onAuthStateChange('SIGNED_OUT') → return early (ignored)
  → onAuthStateChange('INITIAL_SESSION', session)
  → setUser(session.user) → setTimeout(() => fetchProfile(...), 0)
  → ProtectedRoute: profile.subscription_status = 'active' → Dashboard ✓
```

### Fix — 3 changes to `AuthProvider.tsx`

**1. Guard against transient `SIGNED_OUT`:**
```typescript
if (event === 'SIGNED_OUT') return;
```
Explicit logout is already handled by `signOut()` in `authStore.ts`, which calls `supabase.auth.signOut()` and directly clears all state — so this guard does not break the logout button.

**2. Handle `INITIAL_SESSION` for profile fetching:**
```typescript
if (session?.user && (event === 'SIGNED_IN' || event === 'TOKEN_REFRESHED' || event === 'INITIAL_SESSION')) {
```
When Stripe redirects back, Supabase emits `INITIAL_SESSION` (not `SIGNED_IN`) to restore the existing session. Without this, the profile would not be fetched after redirect.

**3. `setTimeout` around `fetchProfile`:**
```typescript
setTimeout(() => fetchProfile(session.user.id), 0);
```
Calling Supabase client methods (DB queries) directly inside `onAuthStateChange` can cause internal deadlocks. Deferring to the next tick safely exits the auth callback before the DB call runs.

## Known Remaining Issue

After the checkout success flow, the user may occasionally be redirected to `/auth` (logged out). This is suspected to be a second wave of `SIGNED_OUT` events firing after the `INITIAL_SESSION` recovery — potentially triggered by `fetchProfile`'s DB call causing an internal auth state re-evaluation in Supabase.

**Impact:** Low — the checkout flow itself (payment confirmed, subscription activated, credits allocated) works correctly. The user can simply log back in and will find their subscription active.

**Priority:** Low — can be revisited in a future session with more detailed event logging.

**Suspected fix path:** Add a `hasInitialized` ref flag in `AuthProvider` so that `SIGNED_OUT` is only ignored during the initial session recovery window, not indefinitely.
